# Dig制度 システム 要件定義書 v1.0

- **作成日**: 2026-07-16
- **対象**: dgloss OS v2.0 / 画面20「Dig評価システム」
- **元資料**: `dig_v5.xlsx`（現行運用Excel・計算ロジックの正）、ワイヤーフレーム `https://dgloss-os-wireframe.vercel.app/`（将来UI）
- **位置づけ**: 現行はExcel運用。これをdgloss OSのアプリ（評価レイヤー）としてシステム化する。

---

## 1. 背景・目的

### 1.1 背景
ディグロス（dgloss）では、営業メンバーの成果を社内通貨「**Dig**」で可視化し、評価・インセンティブ・昇降級に反映する「**Dig制度**」を運用している。現状は `dig_v5.xlsx`（複数シート・数式連携）で手運用しており、以下の課題がある。

- 月初のメンバー情報コピー（値貼付）、月末の実績入力など**手作業が多く属人的**。
- 事業部ごとに異なる売上計上ルール（AIテレアポ／アポプロ／PT）が**数式に埋もれ、可読性・監査性が低い**。
- SFA（D-ONE）・売上管理表・jinjer（人事給与）と**手動連携**で、二重入力・転記ミスのリスク。
- 借入（Digloss Bank）の返済スケジュール・退社時精算が**Excelでは追いにくい**。

### 1.2 目的
- Dig制度の計算・評価・インセンティブ算定を**システム上で自動化**し、手作業と転記ミスを排除する。
- 事業部別の売上計上ルールを**設定として外出し**し、非エンジニアが保守できるようにする。
- SFA・売上管理表・jinjer との**API連携**でデータフローを一本化する。
- 各メンバーが「**マイクロ起業家**」として自分のP/L（予算Dig vs 実績Dig）を把握できるダッシュボードを提供する。

### 1.3 dgloss OS 内の位置づけ
Dig制度は dgloss OS の**評価・タレントレイヤー（画面20）**に属し、上流のアプリからデータを受け取る「下流の集約・評価アプリ」である。

```
CTI(AIテレアポ #07) ─┐
SFA D-ONE(#08) ──────┼→ 売上管理表(#14) ─→ 【Dig評価(#20)】 ─→ 承認ゲート(#16)
jinjer(人事給与API) ─┘         │                  │
                                └ ボーナスDig記録 ─┘  └→ タレント管理(#19)/昇降級
```

## 2. 用語定義（Digの種類）

| 用語 | 定義 | 算出 |
|---|---|---|
| **Dig** | 社内通貨。**1 Dig = 1円相当** | — |
| **予算Dig** | 各メンバーに課される売上目標（コストベース） | 総コスト × 予算係数（既定4.0） |
| **成果Dig** | 営業実績（獲得粗利）から付与される成果報酬 | 事業部別の売上計上ルールで算出 |
| **ボーナスDig** | 営業以外の貢献（学習・ナレッジ共有・組織貢献等）への付与 | ボーナスDig項目マスタ × 記録 |
| **借入Dig** | 会社（Digloss Bank）または相対での貸付 | 借入額 + 利息 |
| **実績Dig** | 評価対象となる合計値 | **成果Dig + ボーナスDig + 借入Dig** |
| **インセンDig（インセンティブ）** | 予算超過分から支給される報奨 | 予算超過分 × 20% |

> ※ **重要**: Digは繰越不可。評価タイミングで回収され0にリセットされる。ただし**借入Digは返済完了まで繰越**。

## 3. スコープ

### 3.1 対象範囲（In Scope）
- 設定マスタ管理（予算係数・利率・共通費・給与レンジ・売上計上ルール・昇降級条件）
- メンバーマスタ管理
- 月次評価（予算Dig／成果Dig／ボーナスDig／借入Dig／実績Dig／達成率／評価ランクの自動計算）
- 事業部別 成果Dig計算エンジン（AIテレアポ／アポプロ／PT事業部）
- ボーナスDig（項目マスタ + 付与記録 + 月次集計）
- 借入管理（Digloss Bank・相対貸借・自動返済・退社時精算）
- メンバー間Dig送金（取引記録）
- 残高計算・インセンティブ算定・評価ランク・昇降級判定
- 四半期／半期の評価サイクルとDigリセット
- 個人P/Lダッシュボード／チームサマリ／ランキング

### 3.2 対象外（Out of Scope・別画面/別システム）
- 売上原データの取得（SFA D-ONE / CTI / 売上管理表 #14 側で管理）
- 給与・勤怠原データ（jinjer 側で管理。APIで取り込むのみ）
- 最終承認ワークフロー（承認ゲート #16 で実施。Dig側は連携のみ）
- コーチング／行動評価（Work Monitor #03・タレント管理 #19。※後述の安全弁参照）

## 4. アクター・権限

| ロール | 権限概要 | 評価サイクル |
|---|---|---|
| **メンバー** | 自分のP/L・実績Dig・借入・取引の閲覧 | 四半期（3ヶ月） |
| **リーダー** | 上記 + 自チームメンバーの閲覧 | 四半期（3ヶ月） |

## 5. 機能要件

### F-1. 設定マスタ管理
現行「設定」シート相当。青文字＝変更可能な設定値として管理する。

| 設定項目 | 既定値 | 用途 |
|---|---|---|
| 対象年月 | 2026-01 | 処理対象月 |
| 月の日数 | 自動（28〜31） | 日割計算 |
| 社会保険係数 | 1.2 | 社会保険・交通費込み |
| 予算係数 | 4.0 | 予算Dig = 総コスト × 係数 |
| 年利率 | 12%（月利1%） | 借入利息 |
| 正社員 共通費（座席代） | 350,000円/月 | 総コスト算入 |
| アルバイト 共通費 | 150,000円/月 | 総コスト算入 |
| 昇降級しきい値 | 120%↑昇2 / 100%↑昇1 / 80%↓降1 / 60%↓降2 | 昇降級判定 |
| 給与レンジ表 | 役職×A/B/Cランクの金額表 | レンジ・役職ベース参照 |
| 売上計上ルール | 事業部別（後述F-3） | 成果Dig算出 |

- **要件**: しきい値・係数・利率・共通費・給与レンジはUIから編集可。編集は監査ログに残す。
- **要件**: 設定変更は**過去確定月に遡及しない**（確定済み評価は値として凍結／後述F-9）。

### F-2. メンバーマスタ管理
現行「メンバー」シート相当。

- 保持項目: 社員ID / 氏名 / 事業部 / 役職 / 職種(FS/IS/CS) / 所属1(グループ長) / 雇用形態(正社員/アルバイト) / 時給 / 出勤日数 / 基本給与 / レンジ / 役職ベース / 入社日 / 退社日 / 評価サイクル(四半期/半期) / ステータス(在籍/退社)
- **要件**: 社員IDはdgloss OS共通のPerson ID（採用→育成→運用で一貫）と一致させる。
- **要件**: アルバイトは「時給 × 出勤日数」で給与を、正社員は基本給与を使用。
- **要件**: jinjer API から給与・勤怠・入退社日を取込み（手入力も可能なフォールバック）。

### F-3. 成果Dig計算エンジン（事業部別売上計上ルール）
成果Dig（＝獲得粗利ベース）は事業部で計上ルールが異なる。**ルールは設定として外出しし、切替可能にする。**

**① AIテレアポ事業部**
- 回線単価: 50,000 Dig／コール数単価: 50,000 Dig
- 契約期間1（POC含む）: 6ヶ月／契約期間2: 12ヶ月
- 計上比率: FS 5割 / IS 5割 / 自アポ自営の場合 10割

**② アポプロ**
- 初回発注額を 1円 = 1Dig で計上（※千円単位は切り捨て）
- 問い合わせもFSに100%計上

**③ PT事業部（暫定）**
- 〇固定報酬（※稼働工数の6割以上をさく場合のみ計上可）: 一人区あたり売上を100%計上 + KPI達成率に応じ残額を計上（100%/80%/60%）
- 〇成果報酬（売上100%で計上）: FS 5割 / IS 5割 / 自アポ自営 10割

> **要件**: 各事業部ルールをパラメータ化した「計上ルール定義」として保持。獲得粗利 → 成果Dig への変換を職種(FS/IS/CS)・自アポ自営フラグで按分する。**PT事業部は暫定のため要確定（§12 論点1）**。

### F-4. ボーナスDig
現行「ボーナスDig項目」「ボーナスDig記録」シート相当。

- **項目マスタ**: 項目ID / カテゴリ / 項目名 / 付与Dig / 上限Dig/月 / 説明 / 有効(ON/OFF)
  - カテゴリ: インプット・アウトプット / スキルアップ / ナレッジ共有 / 改善提案 / 健康・勤怠 / 組織貢献
  - 例: 本の寄付(1,000/上限10,000)、読了アウトプット(1,000/上限3,000)、月間MVP(50,000)、シフト通り出勤(10,000)
  - **要件**: 項目の追加・削除・有効/無効切替をUIから可能に。
- **記録**: 対象年月 / 記録日 / 社員ID / 項目ID / 獲得Dig / 備考（氏名・項目名は自動参照）
  - **要件**: 月上限（上限Dig/月）を超える付与は自動でキャップ。
  - **要件**: 月次評価の「ボーナスDig」列に `SUMIFS(対象年月, 社員ID)` 相当で自動集計。

### F-5. 借入管理（Digloss Bank）
現行「借入管理」シート相当。

- 目的: 入社時OJT期間の未達→降級を回避するための貸付。
- 借入元: 会社（Digloss Bank）／相対（メンバー間）。
- 会社利率: 年利12%（月利1%）。設定で変更可。
- 保持項目: 対象年月 / 借入社員ID・氏名 / 貸出者(会社=ディグロス or 社員ID) / 借入額 / 月利 / 返済期間(ヶ月) / 累積利息 / 借入残高 / 当月返済 / 返済後残高 / 備考
- 計算ロジック（現行数式準拠）:
  - 利息 = 借入額 × 月利
  - 元利合計 = 借入額 + 利息
  - 当月返済 = 元利合計 ÷ 返済期間
  - 返済後残高 = 元利合計 − 当月返済（翌月の借入残高へ繰越）
- **要件**: 毎月の実績Digから**自動返済**。返済スケジュール（1/12回…完了見込み年月）を表示。
- **要件**: 退社時に借入残高が残る場合、**所属グループ（G）が負担**。同一グループ内の相対貸借は相殺処理。負担割合はグループ長（または部長）が判断。
- **注意**: 現行Excelでは空行が `#DIV/0!` を返している → システムでは返済期間0/未入力をガードする。

### F-6. メンバー間Dig送金（取引記録）
現行「取引記録」シート相当。

- 保持項目: 対象年月 / 取引日 / 支払人ID / 受取人ID / 金額 / 内容 / 備考
- 用途: 業務委託の対価（リスト作成代行、テレアポ業務委託 等）をメンバー間で送金。
- **要件**: 送金は支払人のDig残高を減算し受取人へ加算。取引ログに残高推移（Running balance）を表示。
- **要件**: マイナス残高・自己送金・存在しないIDはバリデーションで拒否（§10）。

### F-7. 月次評価
現行「月次評価{月}」シート相当。**制度のメインシート**。

月ごとに各メンバー1行で以下を保持・計算する。

| 区分 | 項目 | 算出 |
|---|---|---|
| 基本 | 対象Q / 対象年月 / 社員ID / 氏名 / 事業部 / 雇用形態 / 給与 / レンジ / 役職ベース / 入社日 / 退社日 | メンバーマスタから取込（値確定） |
| 在籍 | 在籍日数 | 入社日・退社日と対象月から自動算出 |
| 在籍 | 日割係数 | 在籍日数 ÷ 月の日数 |
| コスト | 座席代 | 雇用形態別共通費 × 日割係数 |
| コスト | 総コスト | 給与 × 社会保険係数 × 日割係数 + 座席代 |
| 予算 | 単月予算Dig | ROUND(総コスト × 予算係数, 10万単位) |
| 予算 | 累計予算Dig | 単月予算Dig × 評価サイクル係数（四半期=3相当、半期=6相当）※現行式は×2/×5で要確認（§12 論点2）|
| 実績 | 成果Dig | F-3 で算出（獲得粗利入力 or SFA連携）|
| 実績 | ボーナスDig | F-4 の月次集計 |
| 実績 | 借入Dig | F-5 の当月借入 |
| 実績 | 実績Dig | 成果Dig + ボーナスDig + 借入Dig |
| 評価 | 達成率 | 実績Dig ÷ 予算Dig |
| 評価 | 評価ランク | S≥1.5 / A≥1.0 / B≥0.8 / C≥0.5 / D<0.5 |

- **要件**: 単月・累計の2系統で達成率・ランクを算出（現行のR〜W列＝単月、X〜AC列＝累計）。
- **要件**: 月初にメンバー情報を「値」として取込み確定（後からマスタを変えても過去評価は不変）。

### F-8. 残高計算・インセンティブ・昇降級
現行「残高計算」シート相当。

- 獲得粗利 = SUMIFS(月次評価の成果Dig, 社員ID, 対象Q)
- 達成率 = 獲得粗利 ÷ 営業目標（予算）
- 残高 = MAX(獲得粗利 − 営業目標, 0) + ボーナスDig
- **インセンティブ = 残高 × 20%**（売上予算の超過分の20%を支給）
- 借入返済 = SUMIFS(借入管理の当月返済, 社員ID)
- 評価ランク = S/A/B/C/D（達成率しきい値は上記と同一）
- **昇降級判定**（設定F-1のしきい値）: 120%↑=2段昇級 / 100%↑=1段昇級 / 80%未満=1段降級 / 60%未満=2段降級。入社月から対象発生・単月累計で評価。アルバイトの時給幅は100円。
- **要件**: 未達メンバーは「人ルート（コーチング／タレント管理#19）」へ自動連携。

### F-9. 評価サイクルとDigリセット（確定処理）
- アルバイト・メンバー・リーダー: 四半期（3ヶ月）評価 → Digリセット。
- マネージャー・部長: 半期評価 → Digリセット。
- **要件**: 評価タイミングでDigを回収し0にリセット。ただし**借入Digは返済完了まで繰越**。
- **要件**: 確定した月次評価データは**イミュータブル（値凍結）**。以後マスタ変更の影響を受けない。

### F-10. ダッシュボード（画面20 UI）
ワイヤーフレーム画面20準拠。

- **個人P/Lカード**: 社員ID/事業部/役職/評価サイクル + 予算Dig・実績Dig・達成率・評価ランクを大きく表示。
- **実績Dig内訳テーブル**: 成果Dig（獲得粗利）/ ボーナスDig / 借入Dig の明細と合計、予算対比、Q末リセット注記。
- **Digloss Bank パネル**: 年利/月利・自動返済・OJT借入額・返済スケジュール（○/12回・完了見込み年月）。
- **取引記録テーブル**: 日付 / 内容 / 増減 / 残高推移（SFA自動計上・MVPボーナス・相対送金・返済）。
- **チームサマリ（残高計算）**: 氏名 / 予算Dig / 実績Dig / 達成率 / インセンティブ / ランク。未達メンバーは人ルート連携注記。
- **評価ルールセクション**: サイクル定義・昇級しきい値・給与レンジ連動の表示。

## 7. 計算ロジック仕様（現行数式の正・移植対象）

| # | 項目 | ロジック（現行Excel数式に準拠） |
|---|---|---|
| 1 | 在籍日数 | `MAX(0, MIN(退社日 or 月末, 月末) − MAX(入社日, 月初) + 1)` |
| 2 | 日割係数 | `在籍日数 ÷ 月の日数(設定C4)` |
| 3 | 座席代 | `IF(正社員, 共通費350k, 共通費150k) × 日割係数` |
| 4 | 総コスト | `役職ベース × 社会保険係数(1.2) × 日割係数 + 座席代` |
| 5 | 単月予算Dig | `MROUND(総コスト × 予算係数(4.0), 100000)` |
| 6 | 累計予算Dig | `単月予算Dig × IF(四半期,2, IF(半期,5, 1))` ※係数は要確認（§12 論点2）|
| 7 | ボーナスDig集計 | `SUMIFS(記録.獲得Dig, 記録.対象年月=当月, 記録.社員ID)` |
| 8 | 実績Dig | `成果Dig + ボーナスDig + 借入Dig` |
| 9 | 達成率 | `IFERROR(実績Dig ÷ 予算Dig, 0)` |
| 10 | 評価ランク | `S:≥1.5 / A:≥1.0 / B:≥0.8 / C:≥0.5 / D:<0.5` |
| 11 | 利息 | `借入額 × 月利` |
| 12 | 当月返済 | `(借入額 + 利息) ÷ 返済期間` → 返済後残高が翌月へ繰越 |
| 13 | 残高（残高計算） | `MAX(獲得粗利 − 営業目標, 0) + ボーナスDig` |
| 14 | インセンティブ | `残高 × 20%` |

> 実装時は上記を単体テスト化し、`dig_v5.xlsx` の実データ（例: 掛端光 予算Dig 4,300,000／実績Dig 1,710,000／達成率39.8%／ランクD）で回帰一致を検証する。

## 9. 運用フロー

**【月初】**
1. jinjer/マスタからメンバー情報を月次評価へ取込（値確定）。
2. 入社日・退社日から在籍日数の自動計算を確認。

**【月末】**
3. 各メンバーの獲得粗利をSFA連携 or 入力 → 成果Dig自動計算。
4. ボーナスDig記録・借入返済・評価ランクを自動計算。
5. メンバー間送金があれば取引記録へ登録。

**【四半期／半期末】**
6. 残高計算・インセンティブ算定 → 承認ゲートへ。
7. 昇降級判定 → タレント管理へ反映。
8. Digリセット（借入Digは繰越）。評価データを凍結。

## 10. 非機能・セキュリティ要件

- **バリデーション**: 全数値入力にマイナス・異常値拒否。文字列に長さ制限。日付は形式・範囲検証。送金は残高範囲・自己送金・存在ID検証。
- **API規約**: 全POST/PUTで入力検証。必須欠落→400 / 未存在→404 / 競合→409。エラーレスポンスにスタックトレースを含めない。
- **監査性**: 設定変更・評価確定・借入承認・送金を監査ログに記録。確定評価はイミュータブル。
- **権限**: ロールベースアクセス制御（§4）。給与・借入等の機微情報は最小権限。
- **技術スタック**: Next.js + TypeScript + Tailwind + PostgreSQL + Prisma。rawSQL禁止（Prisma標準クエリのみ）。`any`型禁止・全APIレスポンス型定義必須。環境変数はサーバサイドのみ（`NEXT_PUBLIC_`に機微情報を置かない）。
- **パフォーマンス**: 一覧系は1,000件で3秒以内。
- **UI**: iPad(768px)対応・タップ48px以上・フォント16px以上。ローディング/エラー/空状態を全画面に実装。ブランドカラー（Primary #FCB900 / Secondary #FF6900・白背景・黒文字）。

## 11. 移行（Excel → システム）

- `dig_v5.xlsx` の各シート（設定/メンバー/月次評価/ボーナスDig/借入管理/取引記録/残高計算）をCSV/取込で初期投入。
- 現行の確定済み月次データ（2025-3Q 等）は「凍結済み評価」として移行。
- 数式は§7の仕様としてサーバ側ロジックへ移植。Excelとの**回帰一致テスト**を必須とする。
- 現行Excelの既知不具合（借入管理の空行 `#DIV/0!`、シート1の暫定計算）は移行時にクレンジング。
