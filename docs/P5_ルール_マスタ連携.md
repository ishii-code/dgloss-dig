# P5 Dig獲得ルール・マスタ編集・keiyaku連携

要件 F-1/F-2/F-3。事業部別Dig獲得ルール、マスタ編集、keiyaku-kanri-next 契約からのDig反映。

## 1. Dig獲得ルール（CalcRule・F-3）
「1契約でどんな契約内容だと何Dig付与するか」を事業部別に定義。
- ルール種別: `回線コール単価`（回線数×unitLine + コール数×unitCall）/ `初回発注1to1`（initial_fee 1円=1Dig・千円切捨）/ `月額基本料金割合`（base_amount×%）/ `固定Dig`。
- `modelKeyFilter`（課金形態）で適用契約を絞る。`active` でON/OFF。
- 画面: **Dig獲得ルール** タブ（一覧＋登録/更新フォーム）。API: `GET/POST /api/calc-rules`。
- ロジック: `@dig/core` `computeContractDig(contract, rule)`（テスト有）。

## 2. keiyaku-kanri-next 連携（契約→Dig反映・F-3）
- 接続: **keiyaku の Supabase 直結**（`KEIYAKU_*` 環境変数で URL/key・投入待ち）。現状はサンプル契約で検証。契約は `Contract`（keiyaku の型をミラー）としてローカルにキャッシュ。
- 帰属: **`ContractAssignment`**（契約→従業員の折半・share%）。初期値は SFA の担当者名から引く想定（`source: sfa`）、**画面で後から修正可能**（`source: manual`）、**複数人の折半**に対応。
- 反映: ルール適用 → 合計Dig → 帰属で按分（`splitDig`・端数は先頭者へ）→ 従業員別に集計（`aggregateSeikaDig`）→ 月次評価の `seikaDig` を更新し達成率/ランク再計算。
- 画面: **Dig獲得ルール** タブ下部（契約カード・折半エディタ・「成果Digに反映」ボタン）。API: `GET /api/contracts?ym=` / `PATCH /api/contracts/[id]/assignment` / `POST /api/contracts/reflect`。

## 3. マスタ編集（F-1/F-2/F-4）
- **従業員マスタ**: 一覧＋登録/更新/削除（`GET/POST /api/members`・`DELETE /api/members/[id]`）。Person ID 手入力。評価データがある従業員は削除不可（409）→ 退社ステータスに変更。
- **設定**: `PUT /api/settings`（予算係数・社会保険・金利・共通費・初回借入等）。
- **ボーナスDig項目**: `POST /api/bonus-items`（upsert）。
- 全変更は監査ログ（AuditLog）に記録。

## 4. 検証（ローカル Postgres）
- ルール3件・契約3件シード。契約Dig計算（K-1001=25万を近藤/堀川で**折半**125k/125k、K-1002=10万→掛端、K-2001=CRM 80万×50%=40万→本間）。
- `POST /api/contracts/reflect` → 3契約→4名の `seikaDig` 更新（UI/curl 双方で確認）。
- 折半エディタで担当者・%変更→`PATCH assignment`保存。従業員マスタ CRUD。
- `pnpm -r typecheck` / core test **41** / `next build` 全 green。
